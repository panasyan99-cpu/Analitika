# Analitika Web 1.8.9

## Версия 1.8.9 beta

В платформу добавлен пятый подраздел **«Заказ поставщику»**. Он предназначен для подготовки двух независимых заказов — по камням и по жемчугу — на основании фактических продаж, рабочих остатков, ручных комплектов и товара в пути.

Внутренняя аналитическая платформа Princess Jewelry теперь включает пять режимов:

- общий анализ продаж;
- сравнение двух периодов;
- склад Baserow;
- продажи и сетевые остатки Sonu;
- формирование заказа поставщику.

## Быстрый запуск

```bash
pip install -r requirements.txt

```

## Новый режим «Заказ поставщику»
### Рекомендации и Limited Order

- Темп считается по фактическим продажам за три месяца; прогнозы исходного файла игнорируются.
- Срок ожидания заказа — два месяца. Если доступный остаток закончится к приходу, система предлагает ещё один двухмесячный объём.
- Рекомендация становится заказом только после кнопки «Согласен с рекомендацией».
- При положительном ТВП автоматический заказ отключён, но доступен ручной дозаказ.
- TT и баланс комплекта учитываются по ориентирам 5 серёг / 3–4 кольца / 2–3 подвески.
- `Limited Order` выгружается отдельным Excel и не смешивается с основным заказом.


1. Загрузите исходный Excel с любым названием.
2. Выберите **Камни** или **Жемчуг**. Это отдельные черновики и отдельные итоговые файлы.
3. Выберите камень/вид жемчуга и категорию комплектов.
4. Проверьте продажи, «Всего остаток», TT, 63, ТВП и прозрачную рекомендацию по каждому изделию.
5. Нажмите «Согласен с рекомендацией», «Изменить количество» или `Limited Order`. Без явного действия рекомендация не попадает в заказ. Каждое изменение сохраняется одновременно в локальный кэш и постоянное S3-совместимое облачное хранилище.
6. Перейдите к размерам колец и распределите количество по размерам 15–24.
7. Скачайте Excel для поставщика.

### Основные правила

- скорость продаж, прогноз и рекомендации из исходного файла игнорируются;
- категория комплекта определяется самым продаваемым изделием, продажи не суммируются;
- после попадания комплекта в категорию показываются все изделия этого же комплекта и нормализованного камня, включая слабые и нулевые позиции;
- положительный ТВП автоматически блокирует рекомендацию, но оставляет ручной дозаказ; отдельного ТВП-фильтра больше нет;
- новый файл с другим содержимым начинает чистый активный заказ; предыдущий заказ не подмешивается и остаётся доступен в истории сохранённых заказов;
- внутри одного исходного Set# изделия сначала разделяются по нормализованному камню; категория, ТВП и ошибки рассчитываются отдельно для каждой каменной части комплекта;
- отображаемый «Всего остаток» = `Всего − 63 − 20 − Princess Hang`;
- 20 полностью скрыт, 63 показывается справочно;
- при наличии колонки NTR2 используется её фактическое значение; расчёт по разнице применяется только к старым файлам без NTR2;
- только положительный ТВП считается товаром в пути; отрицательный ТВП — ошибка;
- рекомендации строятся по собственному темпу `продажи за 3 месяца / 3` и двухмесячному сроку поставки;
- морской и круглый пресноводный жемчуг исключён;
- BSHQ, BSMQ и Emerald HQ исключены как самостоятельно устанавливаемые камни.
- названия камней нормализуются по тем же синонимам и сокращениям, что и в остальных отчётах;
- Green Stones, Other Topaz и Other Stones используются только как блоки навигации: в карточках и Excel всегда выводится конкретный нормализованный камень;
- кубики «Всего остаток», TT и 63 выводятся только при положительном значении; NTR2 в карточке не показывается;
- раздел `<Без комплекта>` классифицируется по каждой позиции отдельно и выводится последним в своей категории;
- для позиций без комплекта выполняется консервативное сравнение фотографий с ручными Set# того же камня и товарной группы.

### Надёжность черновика

Версия 1.8.9 использует приватное S3-совместимое объектное хранилище как постоянный источник истины. Исходный Excel, подтверждённые и ручные количества, Limited Order, размеры колец, отметки сверки, выбранный камень и этап заказа сохраняются после каждого изменения.

После обновления страницы, нового деплоя или открытия приложения на другом компьютере заказ появляется в блоке **«Продолжить сохранённый заказ»**. Приложение скачивает исходный Excel из облака в новый временный runtime и восстанавливает черновик.

Локальный SQLite остаётся только быстрым резервным кэшем. Предыдущие заказы больше не удаляются при загрузке нового отчёта. Для независимой страховки доступен полный ZIP: исходный Excel + выбранные позиции + manifest.

Перед работой обязательно настройте хранилище по инструкции [`ORDER_STORAGE_SETUP.md`](ORDER_STORAGE_SETUP.md). При `required = true` приложение не позволит начать новый заказ без успешной облачной копии.

## Основная структура Sonu

Модуль Sonu полностью перестроен в компактный управленческий отчет из пяти последовательных групп:

1. Серьги.
2. Кольца.
3. Подвески.
4. Браслеты не полный круг.
5. Браслеты полный круг.

В каждом блоке выводятся:

- таблица по камням;
- количество проданных уникальных SKU;
- проданные изделия;
- общий Total продаж в USD;
- количество SKU на остатке;
- общий сетевой остаток в штуках;
- диаграмма по штукам;
- диаграмма по Total продаж;
- рекомендации к заказу.

Рекомендации не используют горизонты 30/45/90 дней. Решение строится по двум управленческим сигналам:

- продажи высокие, а остаток и выбор SKU низкие;
- SKU на остатке мало, но штук много — запас сконцентрирован в оставшихся непродающихся моделях.

Из Sonu удалены разбивка по магазинам и фильтр проб: весь ассортимент поставщика рассматривается как серебряный. Остаток берется один раз на SKU по всей сети, а продажи суммируются по магазинам.

## Общий и сравнительный отчеты

Основной формат содержит иерархию:

`Магазин → Камень → Проба → Номенклатурная группа`.

Глобальный фильтр металла остается доступен в общем анализе, сравнении и складской аналитике.

## Baserow

Складская аналитика выводится одной длинной страницей: обзор, сувениры, касты, позиции, требующие внимания, движение и поставки.

### Каталог браслетов 1.7.6

- `sonunew.xlsx` визуально проверен: 340 из 350 SKU сразу загружены в справочник.
- Классификация хранится одновременно на уровне SKU и модельной семьи.
- Для ручной проверки оставлены только 10 SKU из 9 пограничных модельных семей.
- Мастер ручного разбора показывает одну крупную фотографию на семью; выбор применяется ко всем её каменным вариантам.
- Скрытое распределение 50/50 удалено: неподтверждённые семьи не смешиваются с двумя основными типами браслетов.

### Ручной разбор браслетов 1.7.5

- Вместо длинной таблицы спорных моделей используется кнопка **«Разобрать спорные модели»**.
- Мастер показывает одну модель за раз: крупную фотографию, SKU, камень, продажи и остаток сети.
- Пользователь выбирает **«С затяжкой / центральная композиция»** или **«Без затяжки / полный круг»**.
- Сохранение доступно после разбора всех моделей; ручной выбор имеет наивысший приоритет в следующих отчётах.
- Решения сохраняются в `data/bracelet_classification_overrides.json`, выгружаются в резервный JSON и могут быть загружены обратно после нового деплоя.
- Excel содержит отдельный лист **«Ручные решения»**.

### Формат чисел 1.7.4

- Денежные показатели USD выводятся без копеек.
- Тысячи разделяются по три знака в локальном формате интерфейса.
- В Excel используется формат `# ##0`, без десятичных знаков.


## Последнее обновление 1.8.9

Заказы поставщику перенесены с временного диска Streamlit в постоянное S3-совместимое облачное хранилище. Excel и черновик теперь переживают обновление страницы, redeploy и смену компьютера. Старые локальные заказы автоматически мигрируют, если ещё сохранились, а новый отчёт больше не удаляет предыдущую историю.

## Последнее обновление 1.8.7

Исправлен критический `NameError` при отрисовке комплектов: хеш активного исходного Excel теперь явно передаётся в каждую карточку и поле заказа. Логика полного состава комплектов, очистки нового отчёта и конкретных названий камней сохранена без изменений.
